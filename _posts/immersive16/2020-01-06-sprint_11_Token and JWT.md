# Token
토큰(Token)을 이용한 인증은 현대 웹서비스에서 많이 이용되고 있다. 토큰을 이용하는 이유는 다음과 같다.

**1.Stateless 서버**
Stateless 서버를 이해가 위해서는 먼저 Stateful 서버에 대한 이해가 필요하다.

1. Stateful 서버: 클라이언트에게서 요청을 받을 때 마다, 클라이언트의 상태를 계속해서 유지하고, 이 정보를 서비스 제공에 이용
  * 대표적으로 Session을 이용하는 웹서버
2. Stateless 서버: 반대로 클라이언트의 상태를 유지하지 않고, 서버는 클라이언트로 부터 받은 요청만으로 작업을 처리
  * 클라이언트와 서버의 연결고리가 없기 때문에 확장성(Scalability)가 증가

**2.모바일 애플리케이션에 적합**
Android/iOS 모바일 애플리케이션 개발에서 쿠키를 이용한 인증 방식은 좋지 않을 것이다. 토큰을 이용하면 보안성과 처리 과정에 대한 효율성을 증가시킬 수 있다.

**3.인증정보를 다른 애플리케이션에 전달**
대표적으로 OAuth가 있으며, 페이스북/구글 같은 소셜 계정들을 이용하여 다른 웹서비스에서도 로그인 할 수 있다.

**4.보안**
토큰 기반 인증 시스템을 사용하여 애플리케이션의 보안을 높일 수 있다.(토큰을 사용한다고 해서 무조건 안전한 것은 아니다.)


## 왜 토큰을 사용할까?
기존의 서버 기반인증 시스템에서는 서버측에서 유저의 정보를 메모리/디스크/데이터베이스 등에 담아 기억하고 있어야 한다. 
하지만 이러한 방식은 웹/모바일 웹 애플리케이션의 부흥과 함께 서버의 확장이 힘들다는 단점 등이 있다.

**서버 기반 인증의 단점**
1. 세션
* 유저가 인증을 할 때, 서버는 이 기록을 서버 측인 세션 저장을 한다. 하지만 로그인 중인 유저의 수가 늘어다면 서버의 과부하가 생길 수 있고, 세션을 데이터베이스에 저장한다고 해도 마찬가지로 데이터베이스 성능에 무리를 줄 수 있다.

2. 확장성
* 세션을 사용하면 서버의 부하를 감당하기 위해 서버를 확장해야 하는데, 이는 생각보다 복잡한 과정이다. 

3. CORS
* 웹 애플리케이션에서 세션을 관리 할 때 자주 사용되는 쿠키는 단일 도메인 및 서브 도메인에서만 작동하도록 설계 되어있다. 따라서 쿠키를 여러 도메인에서 관리하는 것은 번거로운 일이다.


위와 같은 서버 기반 인증의 단점들과 달리 토큰 인증에서는 시스템에 더 이상 유저의 인증 정보를 서버나 세션에 담아두지 않는다. 이를 통해 서버 기반 인증의 단점들을 해소할 수 있다.

**토큰의 장점**
1. 무상태(Statelss)이며 확장성(Scalability)이 좋다.
2. 보안성: 쿠키를 전달하지 않으므로 취약점이 사라진다.
3. 활용성: 로그인 정보를 활용할 수 있는 분야가 커진다.
  * facebook, google 계정으로 다른 웹서비스에서 활용 가능
4. 여러 플랫폼 및 도메인
  * CORS와 관계가 있으며, 애플리케이션의 규모가 커지고 여러 디바이스를 호환시키고 더 많은 서비스를 제공하게 된다.
  * 토큰을 사용하면 어떤 디바이스, 도메인에서도 토큰의 유효성만 검증한다면 요청이 정상적으로 처리된다. 단지 서버측에서 애플리케이션 응답에 다음 헤더만 포함시키면 OK

```javascript
Access-Control-Allow-Origin: *
```
